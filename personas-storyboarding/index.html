Personas & Storyboarding

Introduction

To explore the use of personas and storyboarding in UX research, I chose to investigate how teaching assistants (TAs) use Gradescope for grading.

The Interface

Gradescope is a publicly-available, web-based grading tool used to administer coursework. It allows instructors to set up assignments—comprising questions and rubrics with point values—and collect student submissions. TAs can then view and grade these submissions.

Pictured in the images above are the main interfaces which a TA might interact with during grading. In particular, the second image shows a screen which allows TAs to score students' submissions on a per-question basis, checking rubric items to add or subtract points from the submission.

Not pictured are the numerous keyboard shortcuts which can be used to perform the same tasks as most of these UI elements.

Observations

To begin to develop my personas, I watched 10 TAs grade assignments using Gradescope—some in a group setting, and others by themselves. Here's what I observed:

  • There was a 50-50 split between TAs who graded question-by-question, and TAs who graded submission-by-submission. This split was along course lines.
  • Most users clicked on visible UI elements to navigate, instead of using keyboard shortcuts. However, almost all users used keyboard shortcuts to check rubric items.
  • Users rarely had to navigate to a previous submission; but when they did, especially when several TAs were grading in parallel, they took a long time to do so.

I also interviewed 3 of the 10 TAs after they were done grading. Summaries of their responses are listed below:

  1. How experienced are you in using Gradescope as a TA?
    • Users A and B were relatively inexperienced, while User C was fairly experienced. Both users B and C had used Gradescope to set up assignments for their courses.
  2. What was your goal in using Gradescope?
    • All three users used it to grade students' work and to check off students at labs. User B, a HTA, spoke about using it to grade "in an unbiased way, anonymously". User C just wanted to be done with grading, and this was faster than grading by hand.
  3. Which specific part(s) of its interface did you use to attempt to achieve that goal?
    • Everyone mentioned the Manage Submissions and Grade Submissions tabs. User A described hitting z (Next Ungraded), then 1 (the first rubric item), to "lock" the assignment when grading in parallel with other TAs. User C also brought up a hidden Submissions button, accessible by hovering over a question title, which gave an overview of all grades for one question.
  4. Could you describe one or more ways in which Gradescope's UI helped or hindered your work?
    • Users A and B complained that it was "sliced the wrong way": it was "meant for" grading one question, not one submission, at a time. All three appreciated the keyboard shortcuts for efficient grading. User A again raised the multiple-TAs-grading-at-once concurrency issue.
  5. If you could change anything about Gradescope to improve your user experience, what would it be?
    • Users A and B would add more settings for assignments: to mitigate the concurrency problem, and to make them per-submission, rather than per-question. User C, meanwhile, would implement more secret keyboard shortcuts, for fun.

Personas

Based on my findings, I constructed two personas to broadly represent the users I observed.

Adept Alice

Adept Alice is representative of a TA who's been around the block a few times, and is familiar with working around Gradescope's many quirks.

Her main gripe with Gradescope is that its system image doesn't mesh with her task of grading submissions one-by-one—its UI actively promotes a different mental model, and an incompatible mode of operation. She's also peeved that there isn't a built-in way to prevent multiple TAs from accidentally grading the same submission, but has long since learned to work around it.

She represents parts of users A and C, but also the select few TAs I observed who primarily used keyboard shortcuts to navigate.

Beginner Bob

Beginner Bob is representative of a TA who's new to grading and the processes it involves.

Due to his inexperience, he is unfamiliar with Gradescope's hidden elements, such as buttons that don't appear until you hover over them, and the many keyboard shortcuts which could improve his efficiency. Bob tends to grade slower, as merely navigating its many screens is a more effortful task for him than it is for Alice. This is exacerbated by the many steps which he must take to do something like returning to submissions that he'd previously graded.

He represents parts of users A and B, as well as some of the TAs I observed who had difficulty with navigation.

Conclusion

This exercise, though small, reminded me of the value of collecting observations and user interviews for identifying the strengths and weaknesses of user interfaces.

If I were to do this again for something like a personal website, I'd expand the scope of data collection by watching and interviewing a wider gamut of users. Hopefully, that would provide me with a more complete distribution of the users who'd interact with my UIs!
